# Sprint 3B — Follow-up source context (blocking bug)

**Status:** Implemented (updated 2026-06-08 — NP-17/NP-18 bug fix)  
**Date:** 2026-06-08  
**Depends on:** [3A.3 handover spike](sprint-3a-3-handover-spike.md), [3B.3 workflow polish](sprint-3b-3-workflow-polish.md)

---

## 1. Problem

**Observed:** Source activity NP-15 had `Description` and `Answer` (note/outcome history). Follow-up NP-16 created via `Source_ID` had only the user-entered follow-up description and an **empty** `Answer`.

**Impact:** Assignee opens the new task without conversation history — looks like data loss and breaks handover workflow before Sprint 4.0.

---

## 2. Investigation

### 2.1 What ABRA desktop “Odovzdať” does

From [Sprint 3A.3 handover spike](sprint-3a-3-handover-spike.md) (DEMO API + seed chain):

| Behaviour | Detail |
|-----------|--------|
| API primitive | `POST /crmactivities` with **`Source_ID`** = predecessor id |
| Records created | **One new** open activity (new `DisplayName` / document number) |
| Source terminal state | **`Status 3` (Odovzdané)** — set automatically on child commit |
| Dedicated handover endpoint | **None** |
| `pmchangestate` | Does **not** create a follow-up activity |

Desktop creates a **new step** in the chain; predecessor remains as closed history. Subjects differ per step in DEMO seed (`NP-2/2006` → `NP-6/2006`).

### 2.2 Which fields Gen copies vs requires in POST

| Field group | Auto-copied via `Source_ID`? | Mobile adapter (before fix) |
|-------------|:----------------------------:|----------------------------|
| `Firm_ID`, `ActivityType_ID`, refs | **No** — must POST | ✓ Copied explicitly |
| Dimensions (`BusTransaction_ID`, …) | **No** | ✓ Copied explicitly |
| `SolverUser_ID` / `ResponsibleUser_ID` | **No** | ✓ From assignee picker |
| **`Description`** | **No** | Only if user typed follow-up text |
| **`Answer`** | **No** | **Never sent** |
| `Subject`, `SheduledStart$DATE` | **No** | User input |
| `ID`, `DisplayName`, `Status` | Regenerated by Gen | Not sent |

**Conclusion:** `Source_ID` is a **structural chain link**, not a text-field inheritance mechanism.

### 2.3 Is context expected only through `Source_ID` chain?

**Partially.**

- **ABRA desktop / Gen:** Chain is navigable via `source_id` on child; each row can hold its own `Description` / `Answer`. Closed steps retain their text; children start as new open rows.
- **Mobile CRM field use:** Assignee works from **one screen** (activity detail). They must see prior notes/outcomes on the **open** follow-up without walking the chain. Relying on `Source_ID` alone is insufficient.

DEMO seed NP chain (`6000000101` → `A000000101`) has empty `Description`/`Answer` on all steps — seed simplicity, not proof that handover should drop context.

### 2.4 Root cause in Mobile CRM

`ActivityCreateService.BuildGenPayload` set:

- `Description` — only when `command.Description` (optional form field) was non-empty
- `Answer` — **never**

After complete + follow-up, source already held merged `Answer` (notes + completion outcome). Child was created without copying it.

---

## 3. Recommendation (implemented)

When creating a follow-up from a source activity:

| Field | Rule |
|-------|------|
| **`Answer`** | **Always copy** full source `Answer` (includes note history and completion outcome after `CompleteAsync` runs first). |
| **`Description`** | User follow-up description if provided; otherwise **inherit** source `Description`. |
| **Chain** | Keep `Source_ID` for native Odovzdané semantics; no change. |
| **UI** | No new fields — backend carries context; optional future: link to source activity on detail. |

**Not in scope:** Walking multi-hop chains to merge all ancestors’ answers (Phase 2). Immediate source only.

## 4. Implementation

### 4.1 `ActivityMapper.ResolveFollowUpTextFields`

```csharp
public static (string? Description, string? Answer) ResolveFollowUpTextFields(
    string? sourceDescription,
    string? sourceAnswer,
    string? userDescription)
```

### 4.2 `ActivityCreateService.BuildGenPayload`

After loading source via GET (post-complete when used from complete flow):

- Resolves text fields via helper
- Adds `Description` and `Answer` to Gen POST body when non-empty

### 4.3 NP-17 → NP-18 regression fix

**Symptom:** Child empty despite source having Description/Answer (reproduced NP-8 → NP-19 on adapter PID 7868).

**Investigation:**

| Check | Result |
|-------|--------|
| Gen accepts `Description`/`Answer` on `POST /crmactivities` | **Yes** — direct Gen spike persists both |
| `ResolveFollowUpTextFields` in source | Present |
| Running adapter loaded new Gen DLL | **No** — `MobileCrm.Adapter.exe` held lock; stale `MobileCrm.Adapter.Gen.dll` |
| Re-fetch vs completed detail | Pass **completed detail** text explicitly to avoid timing/re-fetch gaps |

**Additional fix:** `CreateActivityCommand` gains `SourceDescription` / `SourceAnswer`. Complete flow passes `completedDetail` values immediately after `CompleteAsync`. Diagnostic logs: `[FollowUpContext]`.

**Action:** Restart `MobileCrm.Adapter` after every backend change (`Stop-Process` / rebuild / `dotnet run`).
